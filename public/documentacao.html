1. Visão Geral do Sistema

O Ferramentaria OS é um sistema web de gestão de ordens de serviço desenvolvido para o setor de Ferramentaria da SCiTec Laboratório. Qualquer colaborador da empresa pode abrir um pedido de serviço sem necessidade de login. A equipe da ferramentaria gerencia os pedidos por meio de um painel Kanban com arrastar e soltar.

URL de produção
https://ferramentaria-os.vercel.app

Funcionalidades

  • Painel público (Kanban read-only): Qualquer pessoa acessa e vê o quadro de pedidos sem login.
  • Abertura de pedido: Formulário em /solicitar com campos obrigatórios, urgência, prazo desejado e upload de arquivos.
  • Acompanhamento por token: Link único enviado por e-mail — /acompanhar/[token] — mostra status, histórico e comentários.
  • Edição pelo solicitante: Na página de acompanhamento, o solicitante pode corrigir descrição, telefone e prazo desejado (enquanto não estiver concluído/cancelado).
  • Painel admin (Kanban interativo): Login com e-mail/senha. Arrastar cards entre colunas muda o status. Disponível em /admin/dashboard/kanban.
  • Tabela admin: Visão tabular com filtros, busca e exportação para Excel.
  • Gestão de tipos de serviço: Admin cria/edita/desativa tipos em /admin/dashboard/tipos-servico.
  • Gestão de usuários: Admin cria/edita contas de outros usuários (editor/admin).
  • Anexos: Upload de arquivos na criação do pedido e pelo admin. Armazenados no Supabase Storage.
  • Comentários: Internos (só admin vê) e públicos (visíveis ao solicitante).
  • E-mails automáticos: Confirmação ao solicitante, notificação ao admin no novo pedido, alerta de mudança de status (solicitante + admin), alerta de edição pelo solicitante (admin).

lib/auth.ts — JWT (autenticação admin)

  lib/auth.ts 
import { SignJWT, jwtVerify } from 'jose'
import { JWTPayload } from '@/types'

const getSecret = () => new TextEncoder().encode(process.env.JWT_SECRET!)

export async function criarToken(payload: JWTPayload): Promise<string> {
  return await new SignJWT({ ...payload })
    .setProtectedHeader({ alg: 'HS256' })
    .setIssuedAt()
    .setExpirationTime('8h')
    .sign(getSecret())
}

export async function verificarToken(token: string): Promise<JWTPayload | null> {
  try {
    const { payload } = await jwtVerify(token, getSecret())
    return {
      id: payload.id as string,
      email: payload.email as string,
      nome: payload.nome as string,
      role: payload.role as JWTPayload['role'],
    }
  } catch {
    return null
  }
}

middleware.ts — Proteção de rotas admin

  middleware.ts 
import { NextResponse } from 'next/server'
import type { NextRequest } from 'next/server'
import { jwtVerify } from 'jose'

export async function middleware(request: NextRequest) {
  if (request.nextUrl.pathname.startsWith('/admin/dashboard')) {
    const token = request.cookies.get('ferramentaria_token')?.value
    if (!token) {
      return NextResponse.redirect(new URL('/admin', request.url))
    }
    try {
      const secret = new TextEncoder().encode(process.env.JWT_SECRET!)
      await jwtVerify(token, secret)
      return NextResponse.next()
    } catch {
      return NextResponse.redirect(new URL('/admin', request.url))
    }
  }
  return NextResponse.next()
}

export const config = {
  matcher: ['/admin/dashboard/:path*'],
}

app/api/auth/login/route.ts — Login admin

  app/api/auth/login/route.ts 
import bcrypt from 'bcryptjs'
import { criarToken } from '@/lib/auth'

// POST — valida email/senha → bcrypt.compare → criarToken JWT →
//        set-cookie 'ferramentaria_token' (httpOnly, 8h)
export async function POST(request) {
  const { email, senha } = await request.json()
  const usuario = await supabaseServer.from('usuarios')
    .select('*').eq('email', email).eq('ativo', true).single()

  const senhaValida = await bcrypt.compare(senha, usuario.senha_hash)
  if (!senhaValida) return 401

  const token = await criarToken({ id, email, nome, role })
  // Cookie: httpOnly, secure em prod, sameSite: 'lax', maxAge: 8h
  response.cookies.set('ferramentaria_token', token, { httpOnly: true, ... })
  return { ok: true, usuario }
}

7. Fluxos Principais

Fluxo: Abertura de pedido

  1. Colaborador acessa /solicitar
  2. Preenche o formulário (FormularioPedido.tsx)
  3. POST /api/pedidos → valida com zod → INSERT na tabela pedidos → INSERT historico_status
  4. Se há arquivos: POST /api/pedidos/{id}/anexos para cada arquivo (upload para Supabase Storage)
  5. E-mail de confirmação enviado ao solicitante (Resend)
  6. E-mail de notificação enviado ao admin (Resend)
  7. Formulário mostra número do pedido + link de acompanhamento único

Fluxo: Movimentação de card (admin)

  1. Admin faz login em /admin → cookie JWT definido (8h)
  2. No Kanban (/admin/dashboard/kanban), arrasta card para outra coluna
  3. PATCH /api/pedidos/{id}/status com { status: 'em_andamento', observacao? }
  4. Middleware verifica cookie JWT antes de chegar na rota
  5. UPDATE pedidos, INSERT historico_status
  6. E-mail para solicitante notificando mudança de status
  7. E-mail para admin confirmando a movimentação

Fluxo: Solicitante edita pedido

  1. Solicitante acessa seu link /acompanhar/{token}
  2. Se status ≠ concluído/cancelado, vê o componente EditarPedido
  3. Clica em "Editar pedido", modifica campos desejados
  4. PATCH /api/pedidos/{id}/editar-solicitante com token de acompanhamento no body
  5. API verifica o token (sem necessidade de login), atualiza os campos, registra no histórico
  6. E-mail enviado ao admin listando os campos alterados

Migração para servidores da empresa (guia resumido)

  1. Servidor da empresa: Precisa de Node.js 18+ instalado. Funciona em Windows Server ou Linux.
  2. Build: Rodar npm run build no servidor e iniciar com npm start (porta 3000 por padrão).
  3. Variáveis de ambiente: Criar arquivo .env.production ou configurar no ambiente do servidor (não usar .env.local em produção).
  4. Domínio interno: Configurar proxy reverso (nginx ou IIS) apontando para localhost:3000.
  5. Banco de dados: Pode continuar usando o Supabase cloud, ou migrar para PostgreSQL local usando pg_dump / pg_restore.
  6. E-mails: Pode continuar usando Resend, ou configurar SMTP interno (substituir lib/email.ts por nodemailer).
  7. GitHub: Remover a conta pessoal como owner e transferir o repositório para a organização SCiTec no GitHub.
